# Omiyage Roulette（お土産ルーレット）

## サービス概要
旅行先で「誰に・いくらで・何を買うか」に迷う人のための、お土産選び意思決定支援サービスです。  
**予算** と **カテゴリ（任意）** を選ぶだけで、条件に合う候補から **お土産をランダムに1つ提案** します。  
選ぶストレスと時間を減らし、旅行体験を最後まで気持ちよく完結させます。

### ログイン要否について
- **MVPでは、提案（ルーレット）や候補登録などの主要機能はログイン必須** の想定です。  
- 使い方紹介などのページは、必要に応じて未ログインでも閲覧できる形を検討します（本リリースで調整）。

---

## このサービスへの思い・作りたい理由
旅行や出張のたびに「お土産選び」で時間を使い、結局いつも似たようなものを買ってしまう経験がありました。  
お土産は相手への気遣いが必要な一方で、現地では時間も情報も限られ、意思決定コストが高くなりがちです。  
そこで「迷う工程」を小さくし、選択の負担を軽くすることで、旅行の満足度を下げずに“ちょうど良い一品”へ到達できる体験を作りたいと考えました。

---

## ユーザー層について
### 想定ユーザー
- 20〜30代の旅行者（国内外問わず）
- 出張や帰省などで短時間でお土産を決めたい人
- 優柔不断で、比較検討に時間を取られやすい人

### この層を対象にした理由
- 現地滞在時間が限られ、「選ぶストレス」や「買い忘れ」を回避したいニーズが強い  
- スマホでの短時間操作（モバイルファースト）と相性が良い  
- “迷いを減らす”価値が明確で、利用シーンが具体的に想像できる

---

## カテゴリについて（想定）
お土産候補の登録時に、以下のようなカテゴリを想定しています（MVPでは固定 or ユーザー入力のどちらかで実装予定）。  
- お菓子／スイーツ  
- 食品（惣菜・乾物など）  
- 飲料（お茶・コーヒーなど）  
- 雑貨／日用品  
- 工芸品／地域グッズ  
- コスメ／入浴剤など  
- その他（限定品など）

---

## 「旅行先のお土産情報をどう持ってくるか」について
MVPでは **外部サイトから自動取得はせず**、ユーザーが **事前にお土産候補を登録** しておきます。  
旅行先では、その登録済み候補（必要に応じて地域やカテゴリで絞れる）から条件に合うものを提案する設計です。  
本リリース以降、地域別のテンプレデータ配布などで「登録の手間」を減らす拡張を検討します。

---

## サービスの利用イメージ
1. ユーザー登録／ログイン  
2. お土産候補を登録（名称／価格／カテゴリ／説明／画像など）  
3. 旅行先で「予算（上限）」を入力し、必要なら「カテゴリ」を選択してSTART  
4. 条件に合う候補から、提案結果がモーダルで表示される  
5. 気に入った提案は **提案履歴（Selections）／お気に入り** として保存し、後から見返せる（※本リリースで実装予定）

### ユーザーが得られる価値
- 「何を買うか決める」時間と心理的負担の削減  
- 予算オーバーや無難すぎる選択の回避  
- 旅行中の意思決定を軽くし、本来の体験に時間を使える

---

## ユーザーの獲得について
- SNS（X等）で「旅行×意思決定支援」の文脈でデモ動画を短尺投稿  
- 旅行好きコミュニティでの共有（旅行計画・持ち物・現地Tips系）  
- 旅行直前に刺さる訴求（「帰る直前の5分で決める」など）でLP誘導  
- 個人ブログ／技術記事で開発背景とユースケースを発信（検索流入）

---

## サービスの差別化ポイント・推しポイント
- 汎用ToDoやメモではなく、「お土産を決める」という特定の迷いに特化した意思決定支援  
- 予算（上限）×カテゴリで候補を絞り、そこから“1つに決める”までをワンアクションで完結  
- 単なるCRUDではなく、条件抽出＋ランダム提案というメインロジックが体験の核になっている  
- 待機中の演出（提案が出るまでのUI体験）を用意し、提案結果をイベント的に見せる  
- MVPは「ユーザーが事前登録した候補」から提案し、将来的にテンプレ等で登録負担を下げる拡張が可能

---

## 機能候補
### MVPリリースで実装するもの
- ユーザー登録／ログイン／ログアウト  
- お土産候補のCRUD（登録・一覧・詳細・編集・削除）  
- ルーレット機能（予算上限＋任意カテゴリで1件提案）  
- 提案結果の表示（モーダル等）  
- レスポンシブ対応（モバイルファースト）  
- エラーハンドリング／ローディング表示  

### 本リリースまでに追加したいもの
- 提案履歴（Selections）の閲覧・絞り込み  
- お気に入り（ブックマーク）機能  
- 画像アップロード（外部ストレージ連携）  
- お土産カテゴリの拡張、タグ機能  
- 「贈る相手」など追加条件（上司／家族／恋人 等）  
- おすすめ候補のテンプレデータ配布（地域別など）  
- 提案理由の説明（候補の特徴・予算適合の可視化）

## 使用する技術スタック
### 全体方針
短期でも「サービスとしての体裁」を担保しつつ、基礎要件（Auth/CRUD）＋独自ロジック（ランダム提案）＋技術的工夫（演出/外部連携候補）を一点で示す構成。

### バックエンド
- Ruby on Rails 7（API構成）  
- 認証：Sorcery（ログインAPI／ユーザー情報取得）  
- DB：PostgreSQL  
- 主要API：お土産CRUD、条件抽出＋ランダム提案API  

### フロントエンド
- Nuxt 3（SPA想定）  
- 状態管理：Pinia（認証状態・ユーザー情報の管理）  
- UI：Tailwind CSS + DaisyUI（体裁を最短で整える）

### インフラ（想定）
- API：Render  
- DB：Render（PostgreSQL）  
- フロント：Vercel  

### 使用予定のライブラリ（候補）
- 演出：confetti系ライブラリ（結果表示の体験強化）  
- 通信：Nuxt標準のfetch（ofetch想定）
